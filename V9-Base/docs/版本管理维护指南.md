# 版本管理维护机制

> **版本号**：v2.9.0
> **适用范围**：xmgl项目管理技能体系 V9+
> **更新时间**：2026-05-18

---

## 一、版本号规范

### 1.1 版本号格式

```
版本格式：v主版本.次版本.修订号

示例：v2.9.0
- 主版本(2)：重大架构变更时递增
- 次版本(0)：技能数量变化、层级调整、核心流程变更时递增
- 修订号(0)：文档修正、格式调整、错别字修正时独立递增
```

### 1.2 版本级别定义

| 级别 | 触发条件 | 示例 |
|------|---------|------|
| **主版本升级** | 体系定位重大变化、架构重构、全局特性增加 | V10→V9（去除L3时间信标重生） |
| **次版本升级** | 新增/删除技能、层级调整、核心流程变更 | v2.3→v2.4（新增4个技能） |
| **修订版升级** | 文档修正、格式调整、触发词微调 | v2.4.0→v2.4.1（错别字修正） |

### 1.3 版本号与技能体系对照

| 技能体系版本 | 版本号 | 技能数 | 主要变更 |
|:------------|:------:|:------:|:---------|
| V10 | v10.0.0 | 27→28 | 全员L3时间信标 |
| V9 | v2.9.0 | 27 | 纯优化版（从V10重生，去除信标） |

## 二、快照规则

### 2.1 快照触发时机

| 触发条件 | 是否必须快照 | 说明 |
|---------|:------------:|:------|
| 主版本升级（X.0.0） | ✅ 必须 | 重大变更，需完整归档 |
| 次版本升级（Y.0.0） | ✅ 必须 | 功能变更，需记录状态 |
| 修订版升级（Y.Z.0） | ⚠️ 可选 | 文档修正，可选择性归档 |

### 2.2 快照目录结构

```
VERSIONS/
├── V10-M29-L3信标全员版/
│   ├── V10-Base/              ← 5个核心技能
│   ├── V10-Standard/          ← 19个技能
│   └── V10-Full/              ← 28技能+文档套件
├── V9-M29-纯优化版/
│   ├── V9-Base/               ← 5个核心技能+文档套件
│   ├── V9-Standard/           ← 20个技能+文档套件
│   └── V9-Full/               ← 27技能+文档套件
└── ...
```

### 2.3 快照质量要求

| 要求项 | 标准 |
|-------|------|
| **完整性** | 必须包含该层级所有.md文件 |
| **一致性** | 所有文档版本号必须与快照版本一致 |
| **可读性** | 文件编码必须为UTF-8，无乱码 |
| **可回溯性** | 快照路径和版本号必须精确记录 |

## 三、版本检测流程

### 3.1 启动时检测（session-guard联动）

```
新会话启动
    │
    ▼
读取 TRUTH-SOURCE.md 中的体系版本
    │
    ▼
对比 00-技能全景图.md 中的版本号
    │
    ├── 一致 → 正常加载，跳过版本检测
    │
    └── 不一致 → 触发版本漂移警告
                    │
                    ▼
            提示用户执行版本同步
```

### 3.2 版本漂移判定规则

| 检测项 | 判定标准 | 处理方式 |
|-------|---------|---------|
| 版本号不一致 | TRUTH-SOURCE ≠ 技能全景图 | 🔴 阻断性错误，强制同步 |
| 文档数不一致 | 目录 ≠ 历史快照 | 🟡 警告，提示补充/清理 |
| 编码异常 | 文件读取为乱码 | 🔴 阻断性错误，强制从快照恢复 |

## 四、版本升级执行流程

### 4.1 标准升级流程（5步）

```
Step 1: 评估影响范围
    ├── 确认升级类型（主版本/次版本/修订版）
    ├── 清点受影响的文档清单
    └── 评估是否需要创建快照
Step 2: 执行文档更新
    ├── 更新 00-技能全景图.md 版本号
    ├── 更新根目录 README.md 版本号
    ├── 更新各技能 SKILL.md 中的版本号
    └── 更新其他受影响的文档
Step 3: 版本号同步
    ├── 确认所有文档版本号一致
    ├── 更新所有文档的「最后更新时间」
    └── 刷新 CHANGELOG.md
Step 4: 质量验证
    ├── 读取技能全景图，确认无乱码
    ├── 验证所有文档链接有效
    └── 确认版本号一致性
Step 5: 快照归档（仅主/次版本升级）
```

## 五、跳过升级规则

### 5.1 可跳过升级的情形

| 情形 | 是否可跳过快照 |
|-----|:---------:|
| 纯文档修正（错别字、格式） | ✅ |
| 触发词微调 | ✅ |
| 示例更新 | ✅ |
| 链接修正 | ✅ |

### 5.2 不可跳过升级的情形

| 情形 | 是否可跳过快照 |
|-----|:---------:|
| 技能数量变化 | ❌ 必须快照 |
| 层级结构调整 | ❌ 必须快照 |
| 核心流程变更 | ❌ 必须快照 |
| 定位调整 | ❌ 必须快照 |

### 5.3 跳过升级的审计规则

- 跳过升级时，必须在 CHANGELOG 中注明「本版本未创建快照」
- 连续跳过2次修订版升级后，第3次无论何种变更都必须创建快照

## 六、长期未升级项目评估

### 6.1 未升级判定标准

| 未升级时长 | 判定 | 处理方式 |
|-----------|:----:|---------|
| < 30天 | 正常 | 无需处理 |
| 30-90天 | 需要关注 | 提示检查是否需要同步 |
