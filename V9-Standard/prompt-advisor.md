# prompt-advisor

> 🔴 可选插件，不依赖CARL组件，可随时卸载
> 🟢 模式C自动触发，模式A/B需要手动调用

## 三种模式速览

| 模式 | 定位 | 触发方式 | 核心动作 |
|:---|:---|:---|:---|
| **A: 集成审查** | 语义分析+冲突+冗余 | 手动（审查提示词/分析铁律覆盖） | 读取CARL体系 → 扫描 → 输出建议清单 |
| **B: 专家模拟** | 方向性建议 | 手动（提示词建议/优化方向） | 基于历史+记忆给出建议，不读取实际提示词 |
| **C: 智能需求澄清** | 信息完整性检查 | 自动（缺关键要素时）+ 手动 | 意图解析 → 缺口判断 → 输出补充建议 |

## 触发词对照

| 模式 | 触发词 |
|:---|:---|
| A | `审查提示词`、`检查提示词质量`、`分析铁律覆盖` |
| B | `提示词建议`、`优化方向`、`专家视角` |
| C 自动 | 用户直接提问，AI 自动检测信息完整性 |
| C 手动 | `帮我检查提问完整性`、`信息足够吗` |

## 模式C执行流程（核心）

```
用户提问 → 意图解析 → 信息完整性检查 → 缺口判断
  ├─ 信息足够 → 直接回答
  ├─ 信息不足 → 生成「补充建议清单」
  └─ 不确定 → 列出推测项，询问用户
```

## 结构完整性审查（五维度）

| 维度 | 检查内容 |
|:---|:---|
| 身份定义 | 是否明确了 AI 的角色与责任边界 |
| 数据分级 | 是否定义了数据敏感性级别与处理规则 |
| 行为约束 | 是否覆盖了完整的禁止行为清单 |
| 评估标准 | 是否定义了判断输出质量的具体标准 |
| 兜底机制 | 是否定义了信息不足时的降级处理 |
