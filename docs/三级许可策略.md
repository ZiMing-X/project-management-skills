# xmgl 三级许可策略

> 本文档为 GitHub 发布前的许可架构设计参考，定义了 xmgl 项目各版本层的许可证选择及目录隔离策略。

---

## 设计目标

通过清晰的目录结构分隔不同许可代码，实现：
1. **社区传播**（L1：宽松许可）
2. **防抄袭保护**（L2：强 Copyleft）
3. **商业变现**（L3：闭源）

---

## 三级结构

```
xmgl（GitHub 公开仓库）
├── README.md            # 总览，说明三层关系
├── LICENSE              # 默认许可证（L1 Apache-2.0）
│
├── V9-Base/            # L1 基础版（Apache-2.0）
│   ├── LICENSE          # Apache-2.0 全文
│   ├── README.md
│   └── （5个核心技能文件）
│
├── V9-Standard/        # L2 标准版（AGPL-3.0）
│   ├── LICENSE          # AGPL-3.0 全文
│   ├── README.md
│   └── （20个技能文件）
│
└── V8-legacy/          # 旧版参考（CC BY-NC 4.0）
    ├── LICENSE          # CC BY-NC 4.0 全文
    └── （6个技能文件）

（V10-Full 企业版不在此公开仓库中，闭源交付）
```

---

## L1 — V9-Base（Apache-2.0）

**定位**：生态基石，最大化传播

**许可证**：Apache License 2.0

**自由度**：
- 可自由使用、修改、再分发
- 只需保留版权声明
- 适合个人或公司集成到产品中

**目录操作**：
- `V9-Base/LICENSE` = Apache-2.0 全文
- 根目录 `LICENSE` 也放 Apache-2.0（GitHub 默认显示）

---

## L2 — V9-Standard（AGPL-3.0）

**定位**：防止大厂直接复制做商业产品

**许可证**：GNU Affero General Public License v3.0

**核心特性**：
- "网络即分发"：通过网络（SaaS、API 服务）使用该软件，必须将修改也开源
- 有效防止云厂商在不贡献代码的前提下直接商业化
- 与 Apache-2.0 **不兼容**，必须物理隔离

**目录操作**：
- `V9-Standard/LICENSE` = AGPL-3.0 全文
- **绝不直接引用** L2 的 AGPL 代码到 L1 的 Apache 代码中

---

## L3 — V10-Full（商业闭源）

**定位**：商业变现，核心收入来源

**许可证**：商业授权，不公开源码

**交付方式**：
- 付费订阅
- 商业授权协议
- 企业私有部署

**目录操作**：
- 不在公开 GitHub 仓库中建立 `V10-Full/` 目录
- 私下打包交付给企业客户

## GitHub 许可证放置规范

### 物理隔离（核心原则）

每个独立版本目录各自拥有独立的 `LICENSE` 文件，GitHub 许可证检测器按子目录识别。

### 根目录 LICENSE 声明优先权

在根目录 `README.md` 中明确说明各子目录的许可证归属：

> "This license (Apache 2.0) applies to the contents of the `V9-Base/` directory only. Other subdirectories (such as `V9-Standard/`) contain software under their own, independent licenses. For the exact terms of those licenses, please refer to the `LICENSE` file within each respective subdirectory."

### V8-legacy 特殊处理

`V8-legacy/` 使用 `CC BY-NC 4.0`（署名-非商业性使用），`LICENSE` 文件放置 CC BY-NC 4.0 全文。

---

## 许可证兼容性警告

| 组合 | 兼容性 |
|---|---|
| Apache-2.0 → AGPL-3.0 | ❌ 不兼容 |
| AGPL-3.0 → Apache-2.0 | ❌ 不兼容 |
| Apache-2.0 → Apache-2.0 | ✅ 兼容 |
| AGPL-3.0 → AGPL-3.0 | ✅ 兼容 |

**结论**：L1 和 L2 必须物理隔离，代码不能互相引用。
