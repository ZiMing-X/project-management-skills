# problem-solution-standard — 关键问题解决方案标准

治理层约束。将关键问题（P0/P1级）标准化记录并固化为规范。核心原则：**问题分类 → 根因分析 → 解决方案 → 规范固化**。

## 问题分类

| 等级 | 定义 | 处理要求 |
|:---:|:---|:---|
| 🔴 P0 | 直接阻断业务，数据丢失风险 | 立即修复，24h内固化规范 |
| 🟠 P1 | 功能受损，影响部分用户 | 1周内修复，相关规范升级 |
| 🟡 P2 | 体验问题，不影响核心功能 | 排期修复，纳入知识库 |

## 文件命名规范

```
{项目别名}/INSPIRATION/关键问题-{问题类型}-{日期}.md
```

示例：`BAJX/INSPIRATION/关键问题-参数类型截断-2026-05-11.md`

## 规范化流程

```
发现问题 → 判断严重度(P0/P1/P2) → 创建问题文档 → 填写核心问题+根因+方案 → 更新state.md
                                         ↓
问题修复确认 → 提取经验教训 → 判断是否更新技能规范 → 更新TRUTH-SOURCE → 关闭跟踪
```

## 质量检查清单

| # | 检查项 |
|:--|:---|
| 1 | 问题命名符合规范 |
| 2 | 包含错误代码和正确代码对比 |
| 3 | 问题链路分析完整 |
| 4 | 解决方案优先级标注清楚 |
| 5 | 验证方法可执行 |
| 6 | 经验教训有代码示例 |
| 7 | P0问题24h内已修复或已知晓 |

## 关联规范

| 规范 | 关系 |
|:---|:---|
| `enforce-validation-protocol` | 校验协议（问题发现前置） |
| `automation-closure-standard` | 自动化闭环（执行后记录） |
| `version-governance` | 版本治理（规范升级） |


> 记录问题时，必须标注 🔴 修复前 / 🟢 修复后 时间点，并在文档中显式对比两个状态。
> 固化规范时，**回溯问题文档的时间线**，确认修复→验证→固化的时间链完整。
