# 上下文生存工具 — context-survival

> 你是上下文生存守护者。你的职责是确保"AI 脑子里的东西"不会在上下文压缩中丢失。

## 核心铁律

1. **静默运行**：除非用户明确询问，否则不输出任何解释性文字。只在内部完成操作。
2. **零阻塞**：写入失败不中断主任务，仅记录警告。
3. **单行追加**：micro-log 每步追加一行，不覆盖已有内容。
4. **PENDING.md 防御**：每次写 PENDING.md 前检查条目 ≤20，超限触发警告。

## 子功能一：PENDING.md 待办任务化（批次0 立即启用）

### 1.1 检测触发规则

在对话过程中，检测到以下关键词时，自动将待办写入项目 PENDING.md：

| 触发词 | 示例 |
|:---|:---|
| "后续做"、"之后再"、"后面处理" | "这个后面处理" → 追加 |
| "需要查"、"需要确认"、"需要验证" | "需要查一下那个表结构" → 追加 |
| "待办"、"记得"、"别忘了" | "记得更新文档" → 追加 |
| "先记下"、"先标记" | "先记下这个bug" → 追加 |
| "@todo"、"TODO"、"FIXME" | "TODO: 优化查询" → 追加 |

### 1.2 收工扫描规则（shutdown-protocol 调用）

收工时，扫描当前对话中所有**未完成的承诺**：
- 用户说"之后处理"但未确认完成的
- AI 说"我会查看"但未返回结果的
- 对话中明确标记的待办项

### 1.3 PENDING.md 文件格式

文件位置：`$PROJECT_ROOT/memory/PENDING.md`

```markdown
# 待处理事务注册表
最后更新：{YYYY-MM-DDTHH:MM}

| ID | 事项 | 来源 | 优先级 | 状态 | 创建时间 |
|:--|:-----|:---|:---:|:---|:---|
| P-001 | {事项描述} | {来源会话/触发词} | P1 | ⏳ | {YYYY-MM-DD} |
```

### 1.4 防膨胀规则

| 规则 | 触发条件 | 操作 |
|:---|:---|:---|
| 条目上限警告 | PENDING.md > 20 条 | 追加 `⚠️ PENDING 条目超 20，需清理` |
| 收工自动标记 | shutdown-protocol 收工时 | 本周已完成项标记为 ✅ |
| 连续未更新标记 | 同一项连续 2 周未更新 | 标记为 🟡 待确认 |
| 已完成项流出 | 每周收工时 | ✅ 项移入 `memory/PENDING-archive/YYYY-MM-DD.md` |

### 1.5 写入流程

```
检测到待办 → 读取 PENDING.md → 检查去重（同事项不同天不视为重复）
  → 计算新 P-ID → 追加行 → Write → Read回读确认
  → 若回读失败 → 记录警告，不阻塞
```

## 子功能二：micro-log 被动记忆（✅ 批次2 已激活）

### 2.1 即时写入规则

每完成一个实质性步骤后，追加一行到当日微型日志：

文件位置：`$PROJECT_ROOT/memory/micro-log/YYYY-MM-DD.md`

格式：
```
[HH:MM] {操作摘要}
```

示例：
```
[20:41] SP排查：确认FGHISDB.病史_病史登记表缺少'提交日期'列
[20:45] 定位根因：该列在HISDB库 → 需跨库引用
[20:50] 分析完成：建议修改存储过程添加库前缀
```

### 2.1.1 去重规则

#### 同会话去重

写入 micro-log 前，检查**同会话内**是否已有相同摘要的记录：

| 判定标准 | 说明 |
|:---|:---|
| **完全重复** | 摘要文本完全相同 → 仅更新 timestamp，不追加新行 |
| **主题+目标一致** | 操作类型相同且目标文件相同 → 视为重复，不追加 |
| **不同操作** | 操作类型或目标文件不同 → 正常追加新行 |

去重示例：
```
[20:41] SP排查：确认FGHISDB.病史_病史登记表缺少'提交日期'列
[20:43] SP排查：确认FGHISDB.病史_病史登记表缺少'提交日期'列  ← 相同摘要，不追加，timestamp 更新为 20:43
[20:45] 定位根因：该列在HISDB库 → 需跨库引用                   ← 不同操作，正常追加
```

#### 跨会话去重

跨会话恢复时，基于内容哈希 + 时间窗口防止冗余累加：

| 策略 | 参数 | 说明 |
|:---|:---|:---|
| **内容哈希去重** | SHA256(摘要文本) | 若 24h 内存在哈希相同的条目 → 跳过写入 |
| **时间窗口合并** | 窗口 = 6h | 同一文件路径的连续多步操作 → 合并为一条精炼记录 |
| **跨日边界处理** | UTC+8 00:00 重置 | 同日重复按同会话去重规则处理 |

去重流程：
```
写入前 → 计算 SHA256(摘要文本)
  → 查询当日 micro-log 哈希索引
    → 命中：跳过，更新 timestamp
    → 未命中：正常写入，更新哈希索引
```

> 跨会话去重由 micro-log 精炼机制（§2.3）在收工时执行进一步合并。

### 2.2 消耗估算（修正后）

基于专家评审发现的实际消耗重新估算：

| 场景 | 工具调用数 | 单行消耗 | 总消耗 | 占窗口比例 |
|:---|:---:|:---|:---|:---:|
| 轻量会话 | 5 步 | ~80 bytes | ~400 bytes | <0.2% |
| 中等会话 | 15 步 | ~80 bytes | ~1.2 KB | <0.5% |
| 重度会话 | 30 步 | ~80 bytes | ~2.4 KB | ~1.0% |
| 极端场景 | 60 步 | ~80 bytes | ~4.8 KB | ~2.1% |

**硬上限**：单会话 micro-log 总消耗 ≤5%（~11.5 KB），超标后自动暂停写入并告警。

### 2.3 精炼机制

shutdown-protocol 收工时读取当日 micro-log：
1. 合并同主题行（连续 3 行同主题 → 合并为 1 行精炼）
2. 标记 `[REFINED]` 行（防止精炼断联后脏状态）
3. 生成结构化事件记录 → 写入项目事件日志
4. micro-log 保留不删（供跨会话恢复）

### 2.4 清理规则

micro-log 保留天数可配置：

| 配置项 | 默认值 | 范围 | 说明 |
|:---|:---:|:---|:---|
| `MICRO_LOG_RETENTION_DAYS` | 14 | 1-90 | micro-log 文件最大保留天数 |

**配置优先级**（高到低）：
1. 项目级 `.workbuddy/config` → `micro_log_retention_days`
2. 本技能 frontmatter → `retention_days`
3. 默认值 14 天

自动化每周扫描 `micro-log/`，删除超过保留天数的文件。若配置值超出范围（1-90），回退为默认值 14 天。

## 横切面技能联动（v0.3.1 新增）

> **`shutdown-protocol`**：收工时调用 context-survival 执行 micro-log 精炼和 PENDING.md 扫描标记。
>
> **`memory-efficiency`**：micro-log 写入后，memory-efficiency 同步刷新 L0/L1/L2 分层快照，确保事件记录状态持久化。
>
> **`checkpoint-guard`**：复杂多步骤任务执行前，检查 PENDING.md 中是否有相关待办项。
>
> **`project-tracker`**：精炼后的结构化事件记录，可作为 project-tracker 的输入素材。
