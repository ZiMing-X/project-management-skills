## 核心操作流程

### 1. 初始化项目骨架

```
输入：项目名称、项目代号（可选）、项目根目录
输出：标准化项目文档骨架
```

**通用项目结构**：
```
根目录/
├── README.md
├── PROJECT.md
├── CHANGELOG.md
├── docs/
│   ├── requirements.md
│   └── architecture.md
└── ops/
    └── OP-README.md
```

---

### 2. 任务完成汇报流程（强制前置检查）

当用户说"M1.2完成了"或类似表述时：

```
⚠️ Step 0: 输出文件存在性检查（强制）

验证该模块的输出文件是否真实存在：

PowerShell:
  if (Test-Path "输出文件绝对路径") { "存在" } else { "阻断" }

若输出文件不存在 → 必须立即终止状态更新，向用户报告阻断原因
```

```
Step 1: 读取 PROJECT.md
Step 2: 精确查找"模块ID"列匹配的行
Step 3: 更新"状态"和"最后更新"列
Step 4: 自动推进"下一个行动"
Step 5: 保存 PROJECT.md
Step 6: 在 CHANGELOG.md 顶部添加日志
Step 7: 向用户报告更新结果
```

**下一个行动自动推进规则**：

| 状态 | 是否纳入候选 |
|------|------------|
| ✅ 完成 | 否 |
| 🔒 阻塞 | 跳过，不报错 |
| ⏸️ 暂停 | 跳过，不报错 |
| 🔒 等待 | 是（正常候选） |
| ⏳ 待开始 | 是（正常候选） |

选择逻辑：
1. 扫描所有"输入"列，以当前完成模块的输出文件为关键字，找出依赖链
2. 被依赖的模块（状态应为`🔒等待`）→ 解锁为`⏳待开始`
3. 若存在多个可启动模块，选择表格中顺序最靠前的
4. 将"下一个行动"更新为该模块

**状态回滚处理**（当用户主动将某模块改回"待开始"时）：
- 扫描所有"输入"列包含该模块输出的下游模块
- 若下游模块状态为`⏳待开始` → 改回`🔒等待`（级联锁定）
- 在CHANGELOG.md记录级联锁定范围

### 3. 生成操作指令模板

当用户说"生成op-01"或类似表述时：

```
输入：操作名称（如"修复门诊类待确认源表"）
输出：ops/op-XX-{操作简述}.md
```

文件内容使用标准操作指令模板格式。

## 项目模板

### PROJECT.md 模块表格标准格式

```markdown
| 模块ID | 任务 | 状态 | 输入 | 输出 | 最后更新 |
|--------|------|------|------|------|----------|
| M0 | 项目初始化 | ✅ 完成 | - | 文档骨架 | YYYY-MM-DD |
| M1.1 | 提取初步元数据 | ✅ 完成 | 34个SP | metadata_raw.json | YYYY-MM-DD |
| M1.2a | 修复待确认源表 | ⏳ 待开始 | raw.json + procedures | raw.json (更新) | - |
```

### CHANGELOG.md 日志格式

```markdown
## YYYY-MM-DD
- 更新模块 M1.2a 状态为 ✅ 完成
- 下一个行动：执行 M1.2b 修复其他待确认源表

## YYYY-MM-DD-1
- 项目骨架初始化完成
```

## 行为准则

| 准则 | 说明 |
|------|------|
| 日期替换 | 所有 `{当前日期}` 必须替换为真实日期 (格式：YYYY-MM-DD) |
| 文件保护 | 创建文件前先检查是否存在，绝不覆盖已有内容 |
| 格式对齐 | 更新 PROJECT.md 表格时，必须保持Markdown表格格式完美对齐 |
| 日志连续 | 在 CHANGELOG.md 顶部插入新日志，保持时间倒序 |
| 及时确认 | 每次文件操作后，输出一行简短确认消息 |
| 操作后登记 | 每次完成实质操作后，更新项目 memory/ 和 CHANGELOG.md |
| 记忆文件自检 | 会话启动时检查 memory/ 目录是否完整（state.md 必须存在） |

## 交付物与注意事项

### 成果说明

本技能的最终交付物是**标准化项目文档骨架**。

### 适用场景

✅ **最佳适用**：
- 新项目启动时快速建立标准化文档骨架
- 多阶段项目的进度跟踪和状态更新
- 任务完成汇报 + "下一个行动"自动推进

❌ **不适用**：
- 正式事件日志和周报（使用独立技能 `project-tracker`）
- 已有固定项目管理工具（Jira、Project等）的团队
- 纯文件/数据处理任务（无项目概念）

### 使用注意事项

| 注意事项 | 说明 |
|---------|------|
| ⚠️ ops模板生成后需审核 | 自动生成的通用ops模板含占位符，使用前必须手动替换 |
| ⚠️ PROJECT.md表格格式脆弱 | 多人编辑后容易错乱，建议状态更新前检查格式 |
| ⚠️ 不支持删除模块 | 技能仅支持添加和更新，不支持删除。若有误需手动编辑 |
| 💡 project-tracker是独立技能 | 轻量变更日志由本技能处理；正式报告请使用`project-tracker` |
